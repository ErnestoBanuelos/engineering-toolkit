# Kata Retrospective — K 5.D.3
## Server-side CI Required Check

**Kata:** K 5.D.3  
**Completed:** 2026-07-22  
**Outcome:** Successful — all K 5.D.3 success criteria satisfied

---

## 1. Kata Summary

K 5.D.3 required implementing a server-side CI check that validates the
integrity of Engineering Assets on every pull request. The kata produced
three artefacts:

| Artefact | Component | Outcome |
|---|---|---|
| `.github/workflows/skill-integrity.yml` | CI/CD trigger | 17 lines; no logic; calls script |
| `scripts/ci/validate-skill.sh` | Gate script | 8 conditions; commented; locally executable |
| `docs/ci.md` | Gate documentation | Purpose, rules, admin step, limitations, verification record |

The gate enforced 8 named conditions protecting two Engineering Assets:
a portable Skill definition and a hot-context bundle.

Local verification was performed for all 8 failure conditions individually
and for the success case. All 8 failure conditions produced exit code 1 with
named failure messages. The success case produced exit code 0 with all
conditions passing.

## 3. Engineering Decisions Made During the Kata

### Decision 1 — All logic belongs in the script, never in the trigger file

**Context:** The CI/CD trigger file could contain the validation logic directly
(as is common in CI pipelines). This would be shorter and avoid a two-file
structure.

**Decision:** All logic in the script. The trigger file is a thin runner —
it checks out the repository and calls the script. No logic in YAML.

**Rationale:** Logic in the trigger file requires the CI platform to run it.
Logic in a script can be run locally with `bash`. Local executability is what
makes failure cases testable before deployment. Without local executability,
a gate can only be verified by pushing to a branch and waiting for CI — which
means the failure case cannot be confirmed until the gate is already deployed.

**Lesson:** The test-before-deploy property is not free. It is a consequence
of where logic lives. Choosing to put logic in the script is a choice to
preserve the ability to verify the gate independently.

---

### Decision 2 — Dependency failures exit immediately; structural failures accumulate

**Context:** When a required file is absent, the script could either exit
immediately or attempt to continue with the remaining checks.

**Decision:** Exit immediately after dependency failures (missing files).
Accumulate all structural failures before exiting.

**Rationale:** A structural check on a missing file produces a confusing
error from the shell, not a useful FAIL message. Immediate exit on missing
files preserves the quality of failure messages. For structural checks,
accumulating all failures before exiting is more useful to the engineer —
they see all problems in one run, not one problem per run.

**Lesson:** Two different failure modes warrant two different behaviours.
The pattern — exit immediately on dependency failure, accumulate on assertion
failure — is a reusable design pattern worth encoding in the template.

---

### Decision 3 — The `--help` flag must not use a hardcoded line range

**Context:** The initial implementation of `--help` used `sed -n '2,22p'` to
print the header comment block. This is fragile: if the header grows, the
range silently truncates.

**Decision:** Replace with `awk '/^#!/{next} /^#/{print} /^[^#]/{exit}'` —
which prints all leading comment lines without a hardcoded range.

**Rationale:** A `--help` flag that silently truncates the help text is
a defect that is invisible until the header changes. The `awk` pattern is
slightly more complex but has no truncation risk.

**Lesson:** Hardcoded line numbers in scripts are a latent defect. Any
reference to a line by number rather than by content will become incorrect
when the file changes. Prefer content-addressed patterns (`awk` matching on
comment markers) over positional patterns (`sed` with line ranges).

---

### Decision 4 — Conditions are named before implementation, not discovered during it

**Context:** There is natural pressure to write the script and "discover" the
conditions as you go. This produces a gate whose condition list matches what
was easy to check, not what was important to check.

**Decision:** List all conditions explicitly before writing any code. Treat
the list as the specification. Implement exactly the listed conditions.

**Rationale:** A gate designed by writing code first tends to check file
existence and obvious structural properties while missing semantic properties
that require more thought to express. Writing the condition list first forces
the question: "What would I most regret missing?" Writing the code first
produces: "What was easy to write?"

**Lesson:** Specification before implementation is ADR-000 Principle P4,
and it applies to Verification Gates as much as to application features.

---

### Decision 5 — Gate bypass paths are limitations, not defects

**Context:** The engineering review identified three bypass paths: direct
push, `[skip ci]` flag, and workflow file deletion. The question arose whether
these should be fixed or documented.

**Decision:** Document them as known limitations. Partial mitigations exist
(branch protection, CODEOWNERS) but none are complete. Claiming complete
protection where none exists is worse than honestly disclosing the limits.

**Rationale:** A security model that claims protection it does not provide
trains engineers to trust it inappropriately. An honest limitations section
trains engineers to understand what the gate actually guarantees and what
it does not. The gate is valuable even with these limitations; acknowledging
the limits does not diminish the value.

**Lesson:** Honest limitations sections are engineering assets. They prevent
false confidence and guide investment in additional mitigations.

## 4. Lessons Learned

### L1 — A gate that cannot be tested locally cannot be trusted before deployment

The separation of logic from trigger file is not a style preference. It is the
property that makes local pre-deployment verification possible. Any gate that
cannot be tested locally must be deployed before it can be verified — which
means its first test is on production-like infrastructure. This is a higher
risk profile than is necessary.

---

### L2 — Testing only the success case produces a false sense of security

Eight failure conditions were tested individually. This is more work than
testing the success case once. It is also the only way to prove that the gate
can detect each specific violation. A gate tested only in the success state
has only proven that it exits 0 when nothing is wrong — not that it exits 1
when something specific is wrong.

The engineering review in K 5.D.3 identified that failure test 6 (compatibility
field removal) initially passed when it should have failed, due to a defective
test fixture (not a defect in the gate itself). This bug was only found because
every failure case was tested individually. Batch failure testing would have
missed it.

---

### L3 — Advisory gates erode into decorative checks

A gate that does not block merging provides informational value only. Over
time, engineers learn that the check failing does not prevent their work from
proceeding. The failure becomes background noise. The gate becomes a decoration.
Promotion to Required Check is not a deployment detail — it is what makes
the gate a gate rather than a dashboard widget.

---

### L4 — The failure message is the gate's most important output

Engineers encounter gate failures at the worst possible time: when they are
ready to merge and want to proceed. A failure message that says "check 3
failed" produces frustration. A failure message that says
`FAIL: frontmatter missing required field 'description'` produces a specific,
actionable next step. Writing failure messages is not secondary work — it is
the primary user experience design of the gate.

---

### L5 — Gates do not self-protect; that limitation must be explicit

It is possible for a pull request to delete the gate itself, bypassing the
check. This is not a gate defect — it is an inherent property of all
self-referential verification systems. The correct response is:
1. Document the limitation explicitly.
2. Mitigate where possible (CODEOWNERS rules).
3. Accept that complete self-protection is not achievable at this layer.

Pretending the limitation does not exist is the actual defect.

## 5. Verification Strategy

The kata used the following verification approach, which is now encoded in
WF-002 as the standard approach:

**For each condition:**
1. Induce a state where only that condition would fail (others intact).
2. Run the gate script.
3. Confirm exit code 1 and verify the FAIL message names the condition.
4. Restore the repository.

**For the success case:**
1. Confirm the repository is fully intact.
2. Run the gate script.
3. Confirm exit code 0 and all PASS messages.

This approach is more labour-intensive than testing a representative subset
of conditions. It is justified because: (a) gate failures are hard to diagnose
in CI without knowing which condition is at fault, and (b) a condition that
has never been tested in a failing state may be silently broken.

The investment in per-condition testing directly produced the discovery of
the test fixture defect in failure test 6 (described in L2 above). That defect
would not have been found by testing a representative subset.
