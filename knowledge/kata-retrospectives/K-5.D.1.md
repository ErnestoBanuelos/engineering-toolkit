# Kata Retrospective — K 5.D.1
## Bootstrap a Layered Context Bundle

**Kata:** K 5.D.1
**Completed:** 2026-07-22
**Repository analyzed:** cloud-operations-readiness-skill
**Outcome:** Successful — all K 5.D.1 success criteria satisfied

---

## 1. Kata Summary

K 5.D.1 required bootstrapping a layered Context Bundle for an existing repository
that had no prior AI instructions. The kata produced four artefacts:

| Artefact | Layer | Outcome |
|---|---|---|
| `CLAUDE.md` | Hot Context | Created — 70 lines; permanent rules only |
| `docs/context/stack.md` | Warm Context | Created — full stack, verified constraints |
| `context/cold/README.md` | Cold Context index | Created — describes what belongs in cold |
| `context/cold/gap-log.md` | Cold Context gap log | Created — 5 genuine gaps recorded |

No existing AI instructions were present in the repository. No legacy instructions
file was required.

## 3. Engineering Decisions Made During the Kata

### Decision 1 — Hot Context boundary: escalation format belongs in CLAUDE.md

**Context:** The escalation block format (Action / Role / Condition / Artefact)
appears in both CLAUDE.md (Hot Context) and the source SKILL.md. A question arose
whether the format was a fact about the project (→ Warm) or a behavioural rule
(→ Hot).

**Decision:** Hot Context. The escalation format is a permanent behavioural constraint.
Its absence would cause an AI session to produce unstructured escalation outputs.
It is not a stack-specific fact.

**Lesson:** The test for Hot Context is: "Would the session produce unsafe or
incorrect outputs without this?" If yes, it belongs in Hot.

---

### Decision 2 — Cold Context boundary: unverifiable rationale belongs in the gap log

**Context:** The 12-item audit checklist in the source repository has no documented
origin. It could have been placed in Warm Context as a fact ("the checklist has 12
items — verified against SKILL.md line 97") or in Cold Context as a gap.

**Decision:** Both. The count (12 items) is a verifiable fact — Warm Context. The
rationale for why 12 items and which standards informed the list — Cold Context gap.

**Lesson:** A single subject may have a verifiable component (the fact) and an
unverifiable component (the rationale). Split them across layers rather than placing
the entire subject in Cold.

---

### Decision 3 — Cold Context does not require a minimum number of genuine gaps

**Context:** The kata specification required "at least three honest knowledge gaps."
However, challenging any gap fabrication harder than meeting the number target.

**Decision:** Record only genuine gaps. Five were found, but the number is not the
goal. If only two genuine gaps existed, the kata would still satisfy the honesty
requirement by stating "only two genuine gaps could be identified."

**Lesson:** The constraint on Cold Context is honesty, not volume. A gap log with
three fabricated gaps is worse than one with two genuine gaps.

---

### Decision 4 — Verification evidence format: inline vs. summary table

**Context:** When documenting verified claims in stack.md, there are two formatting
options: annotate each claim inline at the point of the claim, or collect all
verifications in a dedicated summary table at the end.

**Decision:** Use inline evidence when the document has few claims; use a summary
table when claims are numerous. In the kata, inline evidence was used because it
makes the relationship between claim and evidence immediately visible to a reader.

**Lesson:** Inline evidence improves readability for short documents. A summary
table is better for auditability in long documents. Both are acceptable; the choice
is a documentation style decision.

## 4. Lessons Learned

### L1 — The Hot Context size limit is an engineering discipline, not an aesthetic choice

Keeping CLAUDE.md to one screen (≈80 lines) forces a rigorous distinction between
rules and facts. Every line that exceeds the budget forces a decision: is this truly
a rule, or is it Warm Context content that happened to be convenient to include? The
size constraint is a quality gate, not a formatting preference.

---

### L2 — Unverified claims without explicit labels are documentation defects

In the kata's stack.md, the web framework for the reference service could not be
verified from available artefacts. Without the **Unverified** label, a reader could
act on the claim as if it were confirmed. Explicit uncertainty is more useful than
false confidence. This confirms ADR-009's independence requirement: a claim that
cannot be independently verified must be labelled.

---

### L3 — Gap logs are most valuable when written at the moment of discovery

Cold Context gaps are easiest to identify during the initial Context Bundle bootstrap,
when the engineer is unfamiliar with the codebase and every undocumented assumption
is visible. As an engineer becomes more familiar with a project, gaps become invisible
through habituation. The bootstrap kata is therefore the optimal time to record cold
context — not retrospectively.

---

### L4 — The three-layer model creates useful creative tension

There is natural pressure to put information into Hot Context because it is always
loaded. The layer model creates productive friction: if content can be deferred to
Warm or Cold, it should be. This keeps Hot Context small enough to be useful. A Hot
Context that grows without constraint eventually becomes indistinguishable from a
general documentation dump.

---

### L5 — Checking for existing AI instructions before creating new ones is not optional

The kata specification required preserving any existing AI instructions. In this case,
no prior instructions existed. But the check must always be performed first. Creating
a new CLAUDE.md without checking would silently overwrite existing instructions —
which could remove safety constraints that were deliberately placed there. The check
is a pre-condition, not a courtesy.
