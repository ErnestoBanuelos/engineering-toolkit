# K-5.D.7 Retrospective

## Kata

**K 5.D.7 — Run a supervised AI implementation session.**

## Summary

This kata required identifying the FIRST SLICE task from an approved task
decomposition and implementing exactly that task under full supervision —
with every proposed modification requiring explicit engineer approval before
application, a mandatory midpoint checkpoint, a complete session log, and
a final engineering review.

The kata produced a single specification-layer diff implementing the risk
classification evaluation order and interface contract for a multi-level
severity model, applied to an analysis capability's specification document.
The session generated one proposal, which was approved on first presentation.
One verification gate passed for each Done checklist item. The session log
and checkpoint were produced. No regressions were introduced.

The kata validated that a correctly structured supervised session produces
traceable, independently reviewable evidence of every decision — and that
the supervision protocol imposes negligible overhead when the task
specification is precise and the context was fully loaded before proposing.

## Engineering Decisions Made During This Kata

### ED-1 — The session boundary is the task, not the session duration

**Decision:** A supervised session covers exactly one task from the task
decomposition. When the task is complete, the session ends — regardless of
how much time remains. When the task cannot be completed, the session stops
at a defined boundary and produces a PARTIAL outcome.

**Rationale:** During the kata, the FIRST SLICE task was completed within
the session. The temptation to continue into the next task was present
because the next task (Risk Level Section Formatter) was small and the
context was already loaded. The kata enforced the stopping rule. This
produced a clean session log for T1 alone, with a well-defined interface
contract that the next session can consume without re-reading the current
session.

**Generalised as:** STD-008 Section 8.3 (Never Continue into the Next Task),
WF-006 Stage 8 (Next Slice or Session Close).

---

### ED-2 — Proposals must show full content before the approval signal

**Decision:** A proposal is not visible unless the engineer can read the
complete text of the proposed change before signalling. A summary, a
description, or a diff header is not a visible proposal.

**Rationale:** The kata demonstrated that showing full proposal text — the
complete inserted section with every line — gave the engineer enough
information to detect any discrepancy between the task specification and
the proposal without requiring them to infer what the full change would look
like. When the engineer approves based on a summary, they are approving
what they imagine the change to be, not what the change actually is.

**Generalised as:** STD-008 Section 1.2 (Proposals Are Visible Before They
Are Applied), PB-004 Section 2 (How to Approve Changes).

---

### ED-3 — The checkpoint is a safety valve, not a status report

**Decision:** The mandatory midpoint checkpoint is not a progress report. Its
value is in the Rejected section and the Discovered section, not in the Done
section. An engineer who reads only the Done section is not using the
checkpoint correctly.

**Rationale:** In the kata, the checkpoint had an empty Rejected section and
an empty Discovered section — reflecting a session where the first proposal
was approved as presented. This was correct. But the format of the checkpoint
forces the engineer to actively note "No proposals rejected" rather than
simply omit the section. That active notation is evidence of scrutiny. An
omitted section would leave doubt about whether the review was conducted.

**Generalised as:** STD-008 Section 5.3 (Checkpoint as a Safety Valve),
PB-004 Mistake 2 (Treating the checkpoint as a status report).

---

### ED-4 — Zero rejections requires a stated reason, not silence

**Decision:** A session log that records zero rejections must include a
brief explanation of why — for example, "the first and only proposal was
correct as presented because the task specification was unambiguous." An
empty Rejected Alternatives section without explanation raises a scrutiny
concern.

**Rationale:** In the kata, the proposal was approved on first presentation.
The Rejected Alternatives section records this explicitly: "No proposals were
rejected in this session. Engineer approved the first and only proposal
without modification." This is evidence of sufficient scrutiny, not evidence
of insufficient review. The distinction is whether the section explains the
absence or leaves it unexplained.

**Generalised as:** STD-008 Section 6 (Rejected Alternative Documentation),
PB-004 Mistake 6 (Recording zero rejections without scrutiny).

---

### ED-5 — Verification is independent of the agent

**Decision:** The engineer must read the evidence for each verification gate
independently. The agent's assertion that a gate passes is not the
verification act. The engineer's reading of the cited evidence is.

**Rationale:** In the kata, each gate cited a specific file and line number.
The engineer could read those lines directly. The gate was not "the agent
says this is present" — it was "line 90 of spec.md contains this text." The
difference is meaningful: the agent can assert incorrectly; the file content
does not. Verification grounded in a specific file location is independently
reproducible.

**Generalised as:** STD-008 Section 7.3 (Verification Is Independent of the
Agent), PB-004 Mistake 3 (Accepting "already handled" without evidence).

## Lessons Learned

### L-1 — Full context loading before proposing eliminates the most common source of incorrect proposals

The kata required reading eight context files before making any proposal.
This produced a single correct proposal on the first attempt. The
correlation is not coincidental: when the agent has read the target file
and knows the exact insertion point before proposing, the proposal reflects
the actual file state rather than an imagined one.

The most common failure mode in supervised sessions is a proposal that is
correct in intent but incorrect in execution — wrong file, wrong heading
level, wrong insertion point. Full context loading before proposing
eliminates this class of error.

---

### L-2 — A precise task specification is the primary determinant of session quality

The session produced one proposal, which was approved immediately, because
the task specification described the Output in enough detail that the agent
could formulate a correct change without iteration. The task's Done checklist
was specific, the insertion point was implied by the specification context,
and the interface contract was explicit.

The inverse is also true: a vague task specification produces iteration, and
iteration produces more proposals, more opportunities for divergence, and more
demand on the engineer's time. Precision in the task specification is the
highest-leverage investment before a supervised session.

---

### L-3 — The session log's value is realised by its reader, not its author

The session log was produced as evidence. Its value is not in having been
written — it is in whether an independent reviewer can reconstruct every
decision from it without access to the conversation. A session log that
requires the original conversation to be interpretable has failed its primary
purpose.

This lesson shapes the template design: the session log must be
self-contained. Every section must be populated with enough specific content
that a reviewer reading it in isolation can verify: what was proposed, what
was approved, why, and what evidence supports each Done checklist item.

---

### L-4 — Supervision overhead is proportional to proposal quality, not to session duration

In the kata, one proposal was evaluated and approved. Total supervision
overhead was one approval decision and a read-back verification. This is
the minimum overhead case — and it was achieved because the proposal was
formulated correctly on the first attempt.

The cost of supervision is not the checkpoint, the session log, or the
verification gates. Those are fixed costs. The variable cost is the
proposal-evaluation-rejection-revision loop. Correct proposals are cheap
to supervise. Incorrect proposals are expensive to supervise. The investment
that reduces supervision cost is task specification quality and context
loading completeness — not session time compression.

---

### L-5 — The FIRST SLICE discipline matters most for context-loading order

In the kata, identifying the FIRST SLICE before beginning the session
determined which context files to read. The FIRST SLICE is the task that
produces the primary signal consumed by all downstream tasks. It is also
the task whose context requirements are the smallest subset of the full
project context.

Implementing the FIRST SLICE first is not just about small, safe increments.
It also minimises the context required to begin — which reduces the risk
of the agent operating on incomplete context and producing a proposal that
conflicts with a part of the codebase or specification it has not yet read.

## Patterns Discovered

### Pattern — Propose-Wait-Signal

**Name:** Propose-Wait-Signal

**Context:** Any supervised AI-assisted implementation session.

**Problem:** Without an explicit protocol, agents proceed from proposal to
application in a single step, bypassing the human review window. Engineers
who do not receive the proposal before application cannot evaluate it.

**Solution:** The agent explicitly pauses after presenting the proposal and
waits for an explicit signal before applying. The three-step sequence is:
present the full change, pause, wait for a named signal (Approve / Reject /
Redirect). No intermediate state (ambiguous agreement, silence, continued
conversation) triggers application.

**Consequence:** Every applied change has a recorded approval signal. The
session log can trace every change to an explicit engineer decision.

---

### Pattern — Evidence-Anchored Gate

**Name:** Evidence-Anchored Gate

**Context:** Any verification step in a supervised session.

**Problem:** Verification gates are commonly recorded as assertions: "The
Done checklist item is satisfied." An assertion without evidence cannot be
independently reviewed and cannot be distinguished from a fabricated result.

**Solution:** Every gate records: the claim (what must be true), the evidence
(the specific file, section, and line that confirms the claim), and the
result (PASS or FAIL). The evidence anchor makes the gate independently
reproducible: any reviewer can read the cited location and confirm or
challenge the result.

**Consequence:** Verification is a traceable act, not a conversational claim.
The session log's Verification Gates section is evidence, not narration.

---

### Pattern — Explicit Absence

**Name:** Explicit Absence

**Context:** Any section of a session log or checkpoint that records a
negative result (zero rejections, nothing blocking, no unexpected findings).

**Problem:** Empty sections are ambiguous. An empty Rejected Alternatives
section may mean no proposals were rejected, or it may mean the section was
never written. Reviewers cannot distinguish between a clean session and an
incomplete session log.

**Solution:** Every section that reports a negative result contains an
explicit statement of that absence and the reason. "No proposals were
rejected in this session. [Brief reason.]" is evidence of scrutiny. An
empty section is not.

**Consequence:** The session log is self-certifying. A reviewer reading it
can distinguish between a completed section with a negative result and a
section that was omitted. The session log format enforces this through
template guidance.

## Future Improvements

### FI-1 — Session log completeness linter

An automated check that reads a session log and verifies: (a) all six
required sections are present, (b) every PASS gate has a non-empty evidence
field, (c) the Outcome section is set to COMPLETE / PARTIAL / BLOCKED (not
left as a placeholder). This would be implemented as an Automation asset.

### FI-2 — Rejection rate metric

A metric tracked across sessions: rejection rate = rejected proposals /
total proposals. A rate of 0% over many sessions may indicate insufficient
scrutiny; a rate of > 50% may indicate task specification quality problems.
The metric would be reported in the session log and tracked in the project's
engineering telemetry.

### FI-3 — Session boundary enforcement automation

An automated guard that detects when an agent has applied changes to files
outside the task's declared scope. This would parse the session log's
Ordered Action Log and cross-reference each "apply" entry against the task
scope. Any out-of-scope application would generate a warning.

### FI-4 — Partial session resume protocol

A formalised protocol for resuming a PARTIAL session. The resume protocol
would read the stopping boundary from the prior session log, verify that the
stopping state is still intact in the repository, and populate the context
for the continuation session without re-reading the entire prior session
history.

### FI-5 — Supervision quality score

A structured scoring rubric applied to session logs during review: (1)
proposal visibility (was full content shown?), (2) approval signal clarity,
(3) rejection documentation completeness, (4) evidence quality in
verification gates, (5) session boundary compliance. A session scoring below
a threshold would be flagged for re-review.
