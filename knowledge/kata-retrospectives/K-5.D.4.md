# K-5.D.4 Retrospective

## Kata

**K 5.D.4 — Write a feature specification that survives a fresh-session audit.**

## Summary

This kata required writing a complete feature specification for a new engineering capability
and then subjecting it to an independent audit at Isolation Tier A — conducted in a fresh
session with no carry-over context from the authoring session.

The audit produced three findings. Two were incorporated. One was deferred with a documented
interim proxy. The specification was updated to reflect all resolutions and reached Reviewed
status.

The kata validated that the Engineering Toolkit's specification and audit patterns are
reproducible and that the audit process surfaces genuine implementability gaps — not merely
stylistic preferences.

## Engineering Decisions Made During This Kata

### ED-1 — Specification section order is fixed and enforced

**Decision:** Specifications must contain exactly six numbered sections in a fixed order:
Behaviour, Concurrency, Errors, Boundaries, Integrations, NFR Budget.

**Rationale:** Section order predictability enables automated verification and reduces
reviewer cognitive load. A reviewer who knows the section order can verify completeness
mechanically before reading content.

**Generalised as:** STD-004 Section 1 (Required Sections table).

---

### ED-2 — Acceptance criteria require four mandatory coverage points

**Decision:** Every AC set must collectively cover the primary success case, a boundary case,
a failure case, and a safety/constraint case.

**Rationale:** During the kata, it was observed that ACs written without coverage guidance
tend to cluster around the happy path and miss failure and constraint behaviours. The four
mandatory coverage points emerged from examining what a minimal but complete AC set must
demonstrate.

**Generalised as:** STD-004 Section 2.4 (Coverage Rule).

---

### ED-3 — An audit with zero findings is evidence of inadequate review, not a perfect specification

**Decision:** Every audit must produce a minimum of one finding. An audit that produces zero
findings is treated as a signal that the review was not rigorous enough.

**Rationale:** In practice, a specification produced in a single session by one author
will always have at least one ambiguity, one undefended assumption, or one underspecified
boundary. If zero findings are produced, the most likely explanation is that the reviewer
was not challenging the specification with sufficient rigour.

**Generalised as:** STD-004 Section 8.2 (Required Findings).

---

### ED-4 — Deferred NFR items require an interim proxy

**Decision:** An NFR item that cannot be fully specified at authoring time must be marked
DEFERRED and must include an interim proxy measurement that can be used immediately.

**Rationale:** One audit finding (AUDIT-ODA-03) identified that a token-count NFR target
was not verifiable because no tokeniser was specified. Rather than leaving the NFR unmeasurable,
the kata established the practice of providing an interim proxy (character count) until the
definitive measurement method is confirmed. This keeps the capability measurable at all
times while acknowledging the outstanding decision.

**Generalised as:** STD-005 Section 5 (Deferred NFR Items).

---

### ED-5 — Routing rules must cover partial-input cases, not only the complete-input case

**Decision:** Every routing rule or dispatch condition must explicitly describe its behaviour
for partial inputs, not only for complete inputs.

**Rationale:** Audit finding AUDIT-ODA-02 identified that a routing rule stated the positive
case (both inputs present → route to X) but was silent on the partial case (one input present,
one absent → ?). An engineer implementing the router would make an undocumented assumption.
Routing rules must be closed: every possible input combination must have a defined outcome.

**Generalised as:** STD-004 Section 4.4 (Granularity Rule), and PB-001 Section 2 (Integrations
question set).

---

### ED-6 — The CRITICAL risk tier in a classification scheme must be independently evaluable

**Decision:** Any "maximum severity" classification tier must define its trigger conditions in
terms of directly observable, artefact-level facts — never in terms of undefined terms such
as "policy violations" or "guarantee breaches".

**Rationale:** Audit finding AUDIT-ODA-01 identified that the CRITICAL trigger condition
referenced a "minimum-replica guarantee" that was never defined in the specification. An
engineer implementing the classifier could not evaluate the CRITICAL condition. The general
principle is that every classification tier's trigger must be independently evaluable by an
engineer with access only to the specification and the input artefacts.

**Generalised as:** STD-004 Section 4.4 (Boundary Definition — Granularity Rule).

## Lessons Learned

### L-1 — The audit is more valuable when the reviewer has no prior context

The strongest findings in this kata came from the reviewer approaching the specification
with no memory of authoring intent. When the author knows what was meant, ambiguities are
invisible. When the reviewer has no prior context, the ambiguities become obvious. This
confirms the value of Isolation Tier A and validates the toolkit's independence requirement
(ADR-009).

### L-2 — Boundary sections require an explicit out-of-scope list

Specifications written without guidance tend to describe what a capability does without
describing what it deliberately does not do. The out-of-scope list prevents scope creep
during implementation and makes the specification more auditable. An empty out-of-scope
list should be treated as a warning sign by reviewers.

### L-3 — NFR targets must be defined before the audit, not after

In this kata, one NFR metric (token count) was identified as unmeasurable during the audit
because the measurement tool was unspecified. Had this been identified at authoring time,
the DEFER would not have been necessary. Authors should pre-screen NFR targets against STD-005
before submitting for audit.

### L-4 — A finding resolution is evidence, not a formality

The before/after record for each INCORPORATE finding creates an audit trail that allows a
third party to verify the specification was actually updated as a result of the audit. This
is a minimum evidence requirement per ADR-009. The finding resolution is not bureaucratic
overhead; it is the artefact that makes the review verifiable.

### L-5 — Specifications should define behaviour contracts, not implementation choices

Throughout the kata, there was a recurring temptation to embed implementation decisions into
the specification (specific algorithm choices, specific library names, specific runtime
configurations). These were consistently removed. A specification that describes observable
behaviour is portable. A specification that describes an implementation is not.

## Patterns Discovered

### Pattern — Closed Input Classification

**Name:** Closed Input Classification  
**Context:** Any routing rule, dispatcher, or classifier that maps input states to processing
paths.  
**Problem:** Routing rules written for the happy path (all expected inputs present) silently
leave partial-input cases undefined. Engineers implementing the router make undocumented
assumptions about the partial case.  
**Solution:** State the routing rule as an exhaustive enumeration: for each possible input
combination (complete, partial, empty), state exactly what happens.  
**Consequence:** Routing rules become longer but implementations become deterministic.
The audit finding that produced this pattern changed the spec from one sentence (positive
case only) to four sentences (positive case + partial case + routing boundary).

---

### Pattern — CRITICAL Tier Observability

**Name:** CRITICAL Tier Observability  
**Context:** Any classification scheme with a maximum-severity tier (CRITICAL, P0, Sev-1).  
**Problem:** Maximum-severity trigger conditions are often stated as policy references or
abstract principles ("violates the production contract", "breaches the SLA"). These cannot
be evaluated without knowing the policy or SLA, which may not be in the specification.  
**Solution:** Define every CRITICAL trigger in terms of directly observable artefact fields.
The evaluating engineer must be able to evaluate the condition with only the specification
and the input data.  
**Consequence:** CRITICAL triggers become more verbose but become independently auditable.

---

### Pattern — Interim Proxy for Deferred Metrics

**Name:** Interim Proxy  
**Context:** Any NFR metric whose definitive measurement method depends on a technology
decision that has not yet been made.  
**Problem:** Deferring an NFR metric with no replacement leaves the capability unmeasurable.
"TBD" metrics are silently ignored during implementation.  
**Solution:** When deferring an NFR metric, always provide an interim proxy: a measurement
that can be performed right now using available tools, even if it is less precise than the
target measurement.  
**Consequence:** The capability remains measurable at all times. The deferred item is
tracked and cannot be silently dropped.

## Future Improvements

### FI-1 — Automated completeness check

The specification template contains placeholder tokens (`{{PLACEHOLDER}}`). A simple
script could verify that no placeholder tokens remain in a submitted specification before
it enters the audit stage. This would be implemented as a Verification Gate (STD-003) and
would replace the manual "no placeholders remaining" check in WF-003 Quality Gate 2.

### FI-2 — AC coverage verification guide

The four mandatory AC coverage points (primary success, boundary, failure, safety) would
benefit from a short worked example showing a complete AC set with each coverage point
labelled. This would be added to the examples domain.

### FI-3 — NFR Budget pre-screen checklist

A one-page pre-screen checklist that authors complete before submitting to audit would
catch the most common NFR omissions (qualitative targets, missing measurement methods,
missing reference sizes) at authoring time rather than at audit time. This would reduce
the number of DEFER findings caused by premature submission.

### FI-4 — Multi-reviewer audit protocol

The current audit standard defines a single reviewer. For high-risk specifications (those
affecting multiple teams or governing capabilities with significant safety implications),
a dual-reviewer protocol would strengthen the evidence. This would be defined as a Level 4
governance requirement in a future ADR amendment.

### FI-5 — Specification versioning convention

The current practice of bumping from 1.0.0 to 1.0.1 on audit amendment and from 1.0.x to
1.1.0 on material scope changes is documented in this retrospective but not yet in a formal
standard. A versioning convention for specification documents (as distinct from software
versioning) should be formalised in STD-004 or a companion document.
