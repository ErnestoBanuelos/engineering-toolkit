# K-5.D.6 Retrospective

## Kata

**K 5.D.6 — Decompose a specification or Brownfield Delta into independently
reviewable implementation tasks.**

## Summary

This kata required producing an implementation decomposition from an
approved Brownfield Delta. The scenario was a validated change request
(the CRITICAL risk level formalisation for a drift analysis capability)
that had been reviewed and signed off. The kata produced a complete
Implementation Plan (`plan.md`) and Task List (`tasks.md`) containing:
six implementation components, six tasks (T1–T6), five interface contracts
(seams), a directed acyclic dependency graph, a First Slice identification
(T1), a Highest-Risk task identification (T3), an Engineering Review that
applied all five challenges, and a Verification Report with all rows passing.

The kata validated that the task decomposition discipline surfaces
implementation ordering risks and interface ambiguities that are not visible
in the source document alone, and that the First Slice and Highest-Risk
identification steps are the most valuable and the most commonly skipped.

## Engineering Decisions Made During This Kata

### ED-1 — The First Slice must produce the primary signal, not implement the most visible feature

**Decision:** The First Slice is the task that produces the primary output
signal (flag, token, classification) that all subsequent tasks depend on —
not the task that implements the most visible or the most complete feature.

**Rationale:** During the kata, the natural First Slice candidate was the
risk classification logic task (T1) because it produced the `critical_active`
boolean that gated every downstream behaviour. An alternative candidate was
the acceptance criterion task (T5) because it was the most behaviorally
complete in terms of user-visible outcomes. The kata confirmed that T1 was
correct: it was the smallest change that (a) confirmed the decomposition
direction was sound, (b) unblocked the maximum downstream work, and (c)
produced a diff under 100 lines. T5 was a better verifier, but T1 was the
better unlocker.

**Generalised as:** STD-007 Section 9 (First Slice Philosophy), PB-003
Section 5 (How to Identify the First Implementation Slice).

---

### ED-2 — The highest-risk task is the one that can produce a regression in the common case, not the edge case

**Decision:** The highest-risk task is the task most likely to produce a
regression in the most frequently executed behaviour — not the task that
implements the most complex or novel logic.

**Rationale:** In the kata, T3 (Escalation Block Builder) was identified
as highest-risk, not T1 (Risk Classification Logic) or T5 (Acceptance
Criteria). The rationale was specific: T3 modified the escalation cap —
a constraint that applies to every Drift Report, not only CRITICAL ones.
An error in T3 would produce a regression visible in LOW, MEDIUM, and HIGH
outputs, which are far more common than CRITICAL outputs. A mistake in T1
would produce incorrect CRITICAL outputs, which would be caught immediately
by AC-5. A mistake in T3 would produce incorrect non-CRITICAL outputs, which
could propagate through many uses before being detected.

**Generalised as:** STD-007 Section 10.1 (Definition), Section 10.2 (Risk
Statement Requirement), PB-003 Section 6 (How to Identify the Highest-Risk
Task).

---

### ED-3 — The interface contract is a design decision, not a documentation formality

**Decision:** Every inter-task dependency requires an explicit interface
contract that names the produced value, its type, and the rule that downstream
tasks must follow. The contract is not a documentation convenience; it is the
design decision about what the interface is.

**Rationale:** During the kata, the `critical_active` boolean was defined
as the central interface for Seam 1. The decision to make it a boolean (rather
than re-deriving the risk level in each downstream task) was itself a
design decision with consequences: it meant that downstream tasks could not
accidentally implement their own classification logic. The interface contract
enforced single-source-of-truth. Treating the interface as a documentation
formality would have missed this consequence.

**Generalised as:** STD-007 Section 7 (Interface Contracts), PB-003 Section 2
(How to Identify Seams), Section 3 (How to Discover Natural Interfaces).

---

### ED-4 — The terminal assembly task must be gated by an approval signal, not just a task completion signal

**Decision:** The terminal task (specification version increment or assembly
task) must be gated by an explicit human approval signal, not merely by the
completion of all prior tasks. Task completion is a technical signal.
Approval is a governance signal. They are different.

**Rationale:** T6 in the kata (Spec Version Increment) was gated by both
"T1–T5 completed" and "Engineering Lead approval." The distinction matters
because a specification in the source repository carried an existing escalation
gate ("Never overwrite spec.md after sign-off without explicit approval").
The decomposition must honour that gate. In general, any terminal task that
writes to a governed artefact must require human approval before execution.

**Generalised as:** STD-007 Section 6.1 (Input Section — prior completed
tasks), WF-005 Stage 7 (Approval gate), ADR-010 E5 (Human decisions
are preserved).

---

### ED-5 — The dependency graph must be constructed before the Engineering Review, not after

**Decision:** The dependency graph must be drawn before the Engineering
Review begins, not as part of the review. An Engineering Review conducted
before the graph is drawn cannot challenge cycle absence or identify missing
seams.

**Rationale:** During the kata, the dependency graph was built from the
interface contracts. The process of building the graph revealed that T4
(NFR Vocabulary Validator) depended only on T1 and not on T2 or T3 — a
dependency structure that was not obvious from the task list alone. Had the
graph been drawn after the review, this discovery would have been missed.
The graph is an analysis tool; it must precede the review that uses it.

**Generalised as:** STD-007 Section 8.2 (Dependency Graph — every
decomposition must include a graph), WF-005 Stage 5 (Dependency Graph
as a separate stage before Engineering Review).

---

### ED-6 — The Engineering Review must challenge each acceptance criterion as a First Slice candidate and reject all but one

**Decision:** The Engineering Review challenge "Does the FIRST SLICE implement
only one acceptance criterion?" requires examining every acceptance criterion
in the source document as a candidate and explicitly rejecting all but the
one chosen.

**Rationale:** The challenge is only meaningful when the review examines
multiple candidates, not just confirms that one was chosen. During the kata,
AC-5 was examined as a full-path candidate (it tests the complete CRITICAL
path end-to-end). The review rejected it because AC-5 requires T1 + T2 + T3
together; it cannot be verified from T1 alone. The proof test (PT-1) was also
considered; it was rejected because it tests boundary behaviour (HIGH preserved
on no-PDB) which, while important, is not the primary activation path. The
selection of T1's core trigger as the First Slice was the result of this
explicit rejection process, not a default choice.

**Generalised as:** STD-007 Section 11.2 (Challenging Your Own
Decomposition), PB-003 Section 5 (Common Mistakes in First Slice
Identification), PB-003 Section 9 (Completion Checklist).

## Lessons Learned

### L-1 — Decomposition reveals interface decisions that the source document does not make explicit

The source document defines behaviour. It does not define interfaces. A
specification says "when X is true, output Y." It does not say "the value
of X is produced by Component A and consumed by Components B and C via
a named interface." That is a decomposition decision, and it has
consequences for testing, for review ordering, and for regression risk.

The process of decomposing the kata's delta forced five interface decisions
that were not explicit in the delta. Each decision was a genuine engineering
choice. A decomposition that produces zero interface decisions is probably
a decomposition that has not been done at all.

---

### L-2 — First Slice identification is consistently the step that is hardest to execute correctly

Every experienced engineer intuitively understands "start small." The
discipline of defining the First Slice precisely — not just starting with
a small task but starting with the specific task that unblocks the most
downstream work, implements exactly one AC, and is independently reviewable
— is harder than it appears.

The kata validated that the selection criteria must be applied in order
(Section 9.1 of STD-007) because the "most interesting" task and the
"most unblocking" task are frequently different tasks.

---

### L-3 — The highest-risk task is rarely the most complex task

In the kata, the highest-risk task (T3) was not the most algorithmically
complex task. T1 had the more nuanced classification logic. T3 was simpler:
inject a block, raise a cap. But T3's simplicity was deceptive: its error
surface covered every output from the capability, not just the new CRITICAL
path. Risk is a function of blast radius, not complexity.

This lesson generalises: the highest-risk task is the one whose error would
be the hardest to detect and the widest in impact, not the one that requires
the most lines of implementation.

---

### L-4 — The terminal assembly task must always be the last task and must always be approval-gated

In every decomposition, there is a terminal task that assembles approved
diffs and increments a version or status. This task must always:

1. Depend on all other tasks.
2. Require human approval before it begins.
3. Introduce no new logic (assembly only).

When the terminal task is not cleanly separated, other tasks start
including assembly work in their diffs, making individual tasks harder to
review and the assembly work invisible to governance.

---

### L-5 — Seams should match component interface contracts in plan.md

The interface contracts in tasks.md (seams) should correspond directly to
the interface contracts in plan.md (component interfaces). When a seam is
introduced in tasks.md that has no corresponding interface contract in
plan.md, it is a sign that the plan is incomplete.

During the kata, this cross-check confirmed the plan and task list were
consistent. In general, building plan.md first and then verifying that
every task-level seam traces to a component-level interface is a reliable
consistency check.

## Patterns Discovered

### Pattern — Primary Signal First

**Name:** Primary Signal First

**Context:** Any decomposition where multiple tasks depend on a single
output value produced by one task.

**Problem:** Teams often begin with the most visible output (the task
that produces the user-facing result) rather than the most foundational
output (the task that produces the value everything else depends on).
Starting with the most visible task produces a large First Slice that is
difficult to review independently.

**Solution:** Identify the primary output signal of the change — the
boolean, token, or classification that gates all downstream behaviour.
The task that produces this signal is the First Slice. Every other task
depends on it.

**Consequence:** The First Slice is small and reviewable. It confirms the
core design decision before any downstream work begins. Errors in the
primary signal are caught early, before they propagate.

---

### Pattern — Blast Radius Risk Assessment

**Name:** Blast Radius Risk Assessment

**Context:** Any decomposition that includes a task modifying a constraint,
cap, or rule that applies to multiple output paths.

**Problem:** Engineers naturally assess risk as a function of complexity.
Simple tasks are assumed to be low-risk. But a simple task that modifies a
cross-cutting rule has a larger error surface than a complex task that
modifies an isolated behaviour.

**Solution:** For each task, estimate the blast radius: how many output
paths would be incorrect if this task is implemented with an off-by-one
error in its condition? The task with the largest blast radius is the
highest-risk task.

**Consequence:** The highest-risk annotation is placed on the task that
matters most, not on the task that looks hardest. Reviewers allocate
scrutiny proportionally to actual risk.

---

### Pattern — Seam-Before-Task

**Name:** Seam-Before-Task

**Context:** Any decomposition where tasks must exchange values or formats.

**Problem:** Interface contracts are commonly written after the task list.
When written after, they describe what was planned rather than constraining
what must be planned. Contracts written after the task list are
descriptions, not agreements.

**Solution:** For each pair of adjacent tasks (producer → consumer),
define the seam before writing either task's Output section. The seam
definition constrains the producer's Output section and the consumer's
Input section. Both tasks are written against the seam, not before it.

**Consequence:** The interface is an agreement, not a coincidence.
Downstream tasks cannot misinterpret the upstream contract because the
contract existed before they were written.

---

### Pattern — Approval-Gated Terminal Task

**Name:** Approval-Gated Terminal Task

**Context:** Any decomposition that modifies a governed artefact (a
specification, a versioned configuration, a published contract).

**Problem:** Terminal tasks that modify governed artefacts are sometimes
treated as mechanical steps that follow automatically from the completion
of all other tasks. This bypasses governance.

**Solution:** The terminal task explicitly depends on a human approval
signal in addition to the completion of all prior tasks. The Done criteria
for the terminal task must include "Named approval has been recorded before
this task began."

**Consequence:** Governance gates are visible in the decomposition, not
implicit in the organisation's process. Reviewers can verify that the
gate is present without consulting an external process document.

## Future Improvements

### FI-1 — Task dependency linter

An automated check that reads a tasks.md file and verifies: (a) the
dependency graph is acyclic, (b) every inter-task dependency corresponds
to a defined seam, (c) no task's Done criteria reference a future task.
This would be implemented as an Automation asset and would apply to any
tasks.md conforming to TPL-TD-002.

### FI-2 — Diff size estimator

A tool that reads a source document section and estimates the likely diff
size of the implementing task. The estimate would be based on the word
count of the source section and a calibration table built from completed
katas. This would help engineers identify tasks that are likely to exceed
100 lines before they begin.

### FI-3 — Component-to-task trace matrix

A structured trace matrix that maps every source document item to its
implementing component (in plan.md) and then to its implementing task (in
tasks.md). This provides full traceability from requirement to task. It
would be added as an optional section in TPL-TD-002.

### FI-4 — First Slice selection rubric

A scoring rubric that quantifies the four First Slice criteria: (1) number
of ACs implemented end-to-end, (2) estimated diff size, (3) number of
downstream tasks unblocked, (4) independent reviewability. A task with
the highest score on criteria 3 and 4 and the lowest score on criteria 1
and 2 is the best First Slice candidate. This rubric would be added to
PB-003 Section 5.

### FI-5 — Seam stability tracking

A convention for tracking whether a seam was frozen, modified, or broken
during implementation. When a seam is modified after being frozen, the
change must be recorded and all consumer tasks must be re-reviewed.
This would be formalised in STD-007 Section 7.3 (Interface Freeze Rule).
